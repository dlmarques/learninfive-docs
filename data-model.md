# Data Model

LearnInFive stores users and generated topics in MongoDB. The server uses TypeScript interfaces as the source-level schema documentation.

## MongoDB Databases and Collections

| Database | Collection | Purpose |
| --- | --- | --- |
| `users` | `user` | User profile, topic history, quiz answers. |
| `topics` | `topic` | Public and personalized generated topics. |

## User Document

Source:

- `learninfive-server/types/User.ts`

Shape:

```ts
interface User {
  userId: string;
  csLevel: string;
  goals: string;
  preferences: string;
  topicsToAvoid?: string;
  pastTopics?: PastTopic[];
  answeredQuizzes?: AnsweredQuiz[];
}

interface PastTopic {
  id: string;
  concept: string;
}

interface AnsweredQuiz {
  id: string;
  correctness: boolean;
}
```

Field notes:

| Field | Meaning | Owner |
| --- | --- | --- |
| `userId` | Clerk user id, from token `sub`. | Clerk/client on profile creation. |
| `csLevel` | User-described computer science level. | User. |
| `goals` | User-described learning goals. | User. |
| `preferences` | User-described preferences and fluent skills. | User. |
| `topicsToAvoid` | Optional user-described avoided topics. | User. |
| `pastTopics` | Denormalized list of personalized topics served to this user. | Server. |
| `answeredQuizzes` | Persisted quiz answer correctness by quiz id. | Server. |

Usage:

- `pastTopics` is converted to a comma-separated concept list for OpenAI prompt context.
- `answeredQuizzes` lets the server return `quiz.userAnswer` for an already answered daily topic.

## Topic Document

Source:

- `learninfive-server/types/Topic.ts`

Shape:

```ts
interface Topic {
  id: string;
  concept: string;
  definition: string;
  realWorldAnalogy: string;
  examples: TopicExample[];
  quiz: TopicQuiz;
  date: Date;
  public: boolean;
  userId?: string;
}

interface TopicExample {
  language: string;
  code: string;
}

interface TopicQuiz {
  id: string;
  question: string;
  answers: TopicQuizAnswers[];
  rightAnswer: string;
  userAnswer?: boolean;
}

interface TopicQuizAnswers {
  id: string;
  content: string;
}
```

Field notes:

| Field | Meaning | Owner |
| --- | --- | --- |
| `id` | UUID generated by server for the topic. | Server. |
| `concept` | Topic title/concept generated by OpenAI. | OpenAI/server. |
| `definition` | Main explanation. | OpenAI. |
| `realWorldAnalogy` | Analogy section. | OpenAI. |
| `examples` | Language-specific code examples. | OpenAI. |
| `quiz` | Multiple-choice quiz for the topic. | OpenAI/server. |
| `quiz.id` | UUID generated by server for personalized topics. Public topic quiz ids depend on the model response unless normalized later. | Server/OpenAI. |
| `quiz.rightAnswer` | Answer id that should be considered correct. | OpenAI. |
| `quiz.userAnswer` | Runtime-only field added when an authenticated user already answered. | Server response. |
| `date` | Topic creation date. Used to identify today's topic. | Server. |
| `public` | Whether this is a public daily topic. | Server. |
| `userId` | Clerk user id for personalized topics. | Server. |

## Client Topic Type

Source:

- `learninfive-client/src/types/Topic.ts`

The client mirrors the topic payload but currently omits backend-only fields such as `public` and `userId`. It includes:

- `id`
- `concept`
- `definition`
- `realWorldAnalogy`
- `examples`
- `quiz`
- `date`

The client narrows example languages to:

- `JavaScript`
- `Python`
- `Java`
- `C#`
- `C++`
- `TypeScript`
- `PHP`

## API Response Convention

Most server responses use this shape:

```ts
{
  success: boolean;
  content: unknown;
}
```

Several endpoints add extra fields:

- `exists` on profile lookup endpoints.
- `edited` on edit profile.
- `correct` on quiz answer.

## Data Lifecycle

### Public topic

1. OpenAI generates a topic.
2. Server adds id, date, and `public: true`.
3. Server stores the topic in `topics.topic`.
4. Anonymous and signed-out visitors read this public topic.

Current nuance:

- Public topic creation does not add `quiz.id` in `requestAndSaveNewPublicTopic`; the public quiz id depends on whether the model includes one.

### Personalized topic

1. Server reads the user's profile.
2. Server extracts past topic concepts.
3. OpenAI generates a personalized topic.
4. Server adds id, quiz id, date, `public: false`, and `userId`.
5. Server stores the topic in `topics.topic`.
6. Server appends `{ id, concept }` to `user.pastTopics`.

### Quiz answer

1. Server compares answer id with `quiz.rightAnswer`.
2. For authenticated users, server appends `{ id: quiz.id, correctness }` to `user.answeredQuizzes`.
3. For anonymous users, the server does not persist the answer; the client uses `localStorage`.

## Data Integrity Considerations

Current implementation details that matter for maintenance:

- There are no visible MongoDB unique indexes in this repository.
- The server does not enforce a schema at database level.
- Public and user topics share the same `topics.topic` collection.
- User learning history is denormalized into `users.user.pastTopics`.
- Quiz answers are append-only in the current code. There is no duplicate prevention for the same quiz id.
- Topic uniqueness is prompt-guided rather than database-enforced.
